# M10-03 · 如何使用 Cloud Agent（实战）

> 上手 Cursor Background Agent，以及让 Agent 跑得好的关键技巧

---

## Cursor Background Agent 上手

Cursor 是目前最容易上手的 Cloud Agent 入口。

### 启动方式

```
打开 Cursor
  → Cmd+Shift+P（命令面板）
  → 搜索 "Background Agent"
  → 或直接在 Chat 面板选择 "Background" 模式
```

也可以在 Chat 框右下角切换模式：
```
Normal / Agent / Background Agent
```

### 基本使用流程

```
1. 切换到 Background Agent 模式
2. 输入任务描述
3. 点击发送
4. Cursor 在云端启动环境（~30秒）
5. 你可以关掉 Cursor 去做别的事
6. 完成后收到通知，查看结果
```

## 写一个好任务描述（最关键）

**Cloud Agent 的效果，70% 取决于你的任务描述质量。**

### ❌ 差的描述

```
帮我优化代码

修一下登录的 bug

重构一下项目
```

这类描述太模糊，Agent 会猜测你的意图，结果往往偏差很大。

### ✅ 好的描述模板

```
## 任务背景
[当前状况是什么，为什么要做这件事]

## 具体要求
[精确描述要做什么，包括边界条件]

## 不要改动
[明确告诉它哪些东西不能碰]

## 完成标准
[如何判断任务完成了：测试通过 / 特定输出 / 代码规范]
```

### 实际例子

```
## 任务背景
我们的 Express 项目里，错误处理逻辑散落在各个路由文件中，
风格不一致，有些直接 console.error，有些 return res.status(500)

## 具体要求
统一所有路由文件的错误处理：
- 所有 try/catch 的 catch 块改为调用 handleError(err, res)
- handleError 函数已在 src/utils/errorHandler.js 中定义
- 涉及文件范围：src/routes/ 下的所有 .js 文件

## 不要改动
- handleError 函数本身不需要修改
- 不要改变任何 HTTP 状态码
- 不要修改路由逻辑，只改错误处理部分

## 完成标准
- 所有路由文件中不再有直接的 console.error 或 res.status(500)
- npm test 全部通过
- 创建 PR 并附上修改文件列表
```

## 任务类型与技巧

### 类型一：代码迁移 / 升级

```
适合度：⭐⭐⭐⭐⭐（最适合 Cloud Agent 的场景）

技巧：
- 先让 Agent 输出迁移计划，你 review 后再执行
- 指定测试覆盖要求（"改完后所有单元测试需通过"）
- 分批进行，每次一个模块

示例：
"将 src/api/ 目录下所有用 callbacks 写的异步函数改为 async/await，
改完后跑 npm test，确保所有测试通过"
```

### 类型二：批量修复

```
适合度：⭐⭐⭐⭐⭐

技巧：
- 给 Agent 一个具体的"判断标准"（什么叫有问题）
- 要求它列出修改清单再执行

示例：
"扫描整个项目，找出所有没有 input validation 的 POST 接口，
用 Zod 添加参数校验。参考 src/routes/auth.js 里已有的写法。"
```

### 类型三：生成测试

```
适合度：⭐⭐⭐⭐

技巧：
- 指定测试框架（Jest / pytest / Go testing）
- 要求覆盖正常流程 + 边界情况 + 错误情况

示例：
"为 src/services/payment.js 里的所有 exported 函数
编写 Jest 单元测试，要求：
1. 正常流程各一个测试用例
2. 参数缺失 / 异常输入的边界测试
3. 用 mock 处理外部 API 调用
目标覆盖率 > 80%"
```

### 类型四：代码审查 + 重构建议

```
适合度：⭐⭐⭐

技巧：
- 先让 Agent 输出问题报告，你来决定哪些要改
- 不要让它直接改，先看报告

示例：
"审查 src/ 目录下所有代码，指出：
1. 重复代码（相似逻辑出现超过 3 次）
2. 函数超过 50 行的
3. 没有错误处理的 async 函数
输出报告，不要直接修改代码"
```

## 其他 Cloud Agent 工具对比

### Devin

```
优点：任务规划能力最强，能从产品需求直接落地代码
缺点：价格贵（$500/月起），上手门槛高
适合：复杂的、跨多系统的大型工程任务
```

### GitHub Copilot Workspace

```
优点：与 GitHub Issue 深度集成，从 Issue 到 PR 一键完成
缺点：功能相对基础，适合轻量任务
适合：GitHub 重度用户，从 Issue 驱动开发的团队
```

### Claude Code（命令行）

```
优点：精准可控，支持本地运行，开发者友好
缺点：需要自己管理运行环境
适合：想要更多控制权的开发者
用法：claude --print "帮我重构 auth.py"
```

## 常见坑和注意事项

### 坑 1：任务太大，结果失控

```
❌ "帮我重构整个项目"
✅ "重构 src/auth/ 模块，其他目录不要动"
```

控制范围，每次任务聚焦一个模块或一类问题。

### 坑 2：没有告诉它"不要动什么"

```
Agent 有时会"顺手"改一些它认为需要优化的代码
→ 在任务描述里加 "不要修改 xxx 以外的文件"
```

### 坑 3：盲目相信测试通过 = 功能正确

```
测试通过只代表现有测试通过
Agent 可能会"修改测试来让它通过"（虽然少见但存在）
→ review PR 时注意检查测试文件是否也被修改了
```

### 坑 4：敏感信息泄露

```
代码会上传到 Cloud Agent 的服务器
→ 不要在代码里硬编码 API Key / 密码
→ 确认你的公司政策允许代码上传
```

## Review PR 的正确姿势

Cloud Agent 完成后给你一个 PR，不要无脑合并：

```
1. 先看 diff 统计：改了多少文件，变更量合理吗？
2. 重点看逻辑改动，而不只是格式变化
3. 特别注意：测试文件是否被修改？
4. 跑一遍 CI，确认测试在你这边也通过
5. 小批量合并，每次 review 一个模块
```

---

## 小结

用好 Cloud Agent 的核心公式：

```
清晰的任务描述
+ 明确的边界（不要碰什么）
+ 具体的完成标准
+ 认真的 PR review
= 省时省力的高质量交付
```

Cloud Agent 不是替代你的工具，是**放大你效率的杠杆**。
