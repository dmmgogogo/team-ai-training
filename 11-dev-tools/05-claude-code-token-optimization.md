## 文章核心观点（给团队的 1 分钟版本）

1. **模型没有“接着读”的记忆**，每次都要重新处理完整上下文。  
2. **Prompt Cache 命中时很省钱**，频繁开新会话反而可能更贵。  
3. **缓存失效（尤其大上下文）代价很高**，所以会话管理比“盲目 /clear”更重要。  
4. **最便宜的 Token 是不进上下文的 Token**：少贴大段日志，多给文件路径让工具检索。

## 详细要点整理

## 1) 为什么会“烧配额”

Claude Code 每次请求通常要带三类输入：

- 固定前缀（系统提示、工具定义、项目规则）
- 历史对话
- 当前新消息

会话轮次越多，历史内容越大。如果不命中缓存，每轮都会重复付费。

## 2) 缓存机制是省钱关键

- 缓存命中依赖“前缀一致”
- 同一活跃会话里，前缀更容易保持稳定
- 闲置较久后继续旧会话，可能触发大规模重建

结论：**不是“永远新会话”，也不是“永远一个会话”，而是按任务与活跃度决策。**

## 3) 会话决策原则（可直接执行）

继续当前会话（优先）：

- 任务没变（仍在同一模块/同一问题）
- 会话仍活跃（间隔不长）
- 现有上下文对当前任务仍有价值

开新会话：

- 任务切换（如从认证模块转到支付模块）
- 会话长期闲置后恢复
- 上下文噪音明显增多（大量无关试错记录）

## 4) 慎用超大上下文窗口

- 大窗口不是越大越好
- 对日常改 bug / 小重构，通常并不需要极限上下文
- 大上下文在缓存失效后的重建成本更敏感

## 5) 六条落地建议（团队版）

1. 日常优先使用成本更稳的模型档位，复杂任务再升档。  
2. 避免会话中途频繁切模型（会影响缓存复用）。  
3. 精简项目规则文件，减少每轮固定注入体积。  
4. 能用命令行完成的检索优先命令行，少把长结果直接塞进对话。  
5. 复杂任务先做计划再执行，减少返工轮次。  
6. 用读取权限/路径约束，避免模型扫描无关目录（如 `node_modules`、构建产物）。

---

## 给你团队的“立即可用”操作清单

1. 把“会话切换规则”写进团队规则（任务切换才新开会话）。  
2. 统一日志处理习惯：给路径，不贴万行日志。  
3. 每周复盘一次“高成本会话”，找出冗余上下文来源。  
4. 给项目加目录读取边界，减少无关扫描。
