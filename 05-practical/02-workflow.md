# M5-02 · AI 辅助开发工作流

> 把 AI 真正融入日常开发流程，而不是偶尔用一下。

---

## 完整开发流程中的 AI 切入点

```
需求理解  →  方案设计  →  编码实现  →  调试测试  →  Code Review  →  上线维护
    ↑              ↑            ↑            ↑              ↑              ↑
   AI 帮          AI 帮        AI 帮        AI 帮          AI 帮          AI 帮
  澄清需求        出方案       写/补全      找 Bug         审查           分析日志
```

---

## 阶段一：需求分析与方案设计

### 用 AI 澄清模糊需求
```
产品说："做一个用户积分系统"

问 AI：
"我需要实现用户积分系统，请帮我列出需要考虑的所有细节：
积分规则、消费场景、过期机制、并发处理、数据一致性等。
用问题清单形式输出，我需要和产品确认。"
```

### 用 AI 生成技术方案
```
"基于以下需求，设计积分系统的数据库 Schema 和核心 API：
- 用户完成订单获得积分（订单金额×10）
- 积分可用于抵扣（100积分=1元）
- 积分有效期1年
- 需要完整的积分流水记录
- 并发扣减不能超发
技术栈：FastAPI + PostgreSQL"
```

## 阶段二：编码实现

### 工作流推荐（以 Cursor 为例）

**Step 1：用 Chat 生成框架**
```
在 Cursor Chat 里：
"根据我们的项目结构（@src/），
帮我创建积分系统的基础文件：
- models/point.py（数据库模型）
- api/points.py（API 路由）
- services/points.py（业务逻辑）
```

**Step 2：用 Agent 填充实现**
```
切换到 Agent 模式：
"实现 services/points.py 中的积分发放函数，
要求：事务安全、记录流水、幂等处理（同一订单不重复发积分）"
```

**Step 3：Tab 补全细节**
```
写函数签名 → Tab 接受 → 调整细节 → 继续
```

---

## 阶段三：调试

### Debug 工作流

**方案 A：直接粘贴错误**
```
1. 运行代码，出现错误
2. 复制完整错误栈到 Cursor Chat
3. Cursor 分析原因 + 提供修复
4. 一键接受修改
```

**方案 B：让 AI 帮你加日志**
```
"在这个函数的关键步骤加上日志输出，
帮我定位哪一步出问题了：
[函数代码]"
```

**方案 C：橡皮鸭调试 + AI**
```
"我来解释一下这段代码的逻辑，你帮我找出哪里可能有问题：
[逐步解释你的代码逻辑]"
```

## 阶段四：测试

### 用 AI 生成测试用例

```
"为以下函数生成完整的单元测试，
覆盖：正常情况、边界条件、异常情况：

[函数代码]

使用 pytest，Mock 数据库调用"
```

### 让 AI 想你没想到的测试场景

```
"我的积分扣减函数测试了以下场景：
[你写的测试]

还有哪些边界情况我可能忽略了？"
```

## 阶段五：Code Review

### 提交前自审

```
在 Cursor 里选中所有改动的文件：

"@src/services/points.py @src/api/points.py
请 Review 这些代码，关注：
1. 安全漏洞（SQL 注入、权限绕过）
2. 并发安全（锁、事务）
3. 性能（N+1 查询、缺失索引）
4. 错误处理是否完整"
```

### AI 自动 PR 描述

```
"根据以下 git diff，生成 PR 描述：
- 变更摘要
- 主要改动说明
- 测试验证方式
- 注意事项

[git diff 内容]"
```

## 阶段六：上线与维护

### 分析生产日志

```
"以下是生产环境错误日志，帮我：
1. 识别错误类型和严重程度
2. 可能的根本原因
3. 紧急修复建议

[日志内容]"
```

### 读懂陌生代码

```
"我接手了一个旧系统，@legacy/payment.py
帮我：
1. 解释整体功能
2. 画出主要函数的调用关系（文字版）
3. 指出哪些地方可能有坑"
```

## 高效工作习惯

### 习惯一：始终保持 Git 干净
```
使用 AI 前先 commit
→ AI 改坏了？git checkout .
→ 成本极低，随意尝试
```

### 习惯二：写好 .cursorrules

项目规范写进 `.cursorrules`，让 AI 永远按你的风格写：
```
# 技术栈
- Backend: FastAPI + SQLAlchemy 2.0
- Python 3.11+，用 type hints
- 异步优先，所有 IO 操作用 async/await

# 代码规范
- 所有公开函数写 docstring
- 错误统一用自定义异常类
- 日志统一用 structlog

# 禁止
- 不用 print 调试，用 logger
- 不在 API 层写业务逻辑
```

### 习惯三：迭代而非一次生成

```
❌ 一次性：
"帮我实现完整的积分系统"
→ 生成一堆代码，可能跑偏

✅ 迭代式：
1. "先帮我设计数据库 Schema" → 确认 →
2. "实现积分发放函数" → 确认 →
3. "加上幂等处理" → 确认 →
4. "写测试" → 完成
```

### 习惯四：学会拒绝和修正

AI 建议不好就说：
```
"这个实现用了全局锁性能差，
改成乐观锁（version 字段）重试的方式"

"这个函数太长了，拆成 3 个小函数"

"改用 UUID 主键，不用自增 ID"
```

## 团队协作规范建议

| 场景 | 建议 |
|------|------|
| AI 生成的代码 | 照常走 Code Review，作者对质量负责 |
| AI 写的注释 | 确认准确性，错误注释比没注释更糟 |
| 敏感数据 | 脱敏后再发给 AI（用 `xxx` 替换真实数据）|
| 核心算法 | AI 辅助，但必须理解每行代码 |
| 工具选型 | 团队统一（推荐 Cursor Business 统一账号）|
