# ⭐️ M5-10 · 代码审查与质量控制

> AI 生成的代码不等于正确的代码。学会用 AI 辅助 review，而不是无脑信任，才是正确姿势。

---

## AI 生成代码的常见问题

### 🐛 功能性问题
| 问题类型 | 例子 |
|---------|------|
| **逻辑错误** | 边界条件处理不当、off-by-one 错误 |
| **过时 API** | 使用已废弃的方法或旧版本语法 |
| **幻觉依赖** | 引用不存在的库或方法 |
| **并发问题** | 竞态条件、死锁，AI 经常忽略 |

### 🔒 安全漏洞
| 漏洞 | 描述 |
|------|------|
| **SQL 注入** | 字符串拼接 SQL 而非参数化查询 |
| **XSS** | 未转义用户输入直接渲染 |
| **硬编码密钥** | API Key 写在代码里 |
| **不安全的反序列化** | pickle、eval 等危险操作 |
| **权限控制缺失** | 忘记加认证/鉴权 |

### 📉 质量问题
- 没有错误处理（只有 happy path）
- 没有日志
- 命名不符合项目规范
- 测试覆盖不足
- 注释与实际代码不符

## 如何 Review AI 生成的代码

### 分层审查法

**第一层：能跑起来吗？**
```bash
# 先跑起来再说
python xxx.py
# 看有没有明显报错
```

**第二层：逻辑对吗？**
- 手动 trace 关键路径
- 重点看条件判断、循环边界
- 问自己：极端输入（空值、超大值、特殊字符）会怎样？

**第三层：安全吗？**
- 搜索关键词：`eval`、`exec`、`pickle`、字符串拼接 SQL
- 检查所有外部输入是否有验证

**第四层：符合项目规范吗？**
- 命名风格
- 错误处理方式
- 日志格式

## 用 AI 辅助 Review AI 代码

让 AI 自己找自己的问题：

```
Prompt 模板：

请 review 以下代码，重点检查：
1. 安全漏洞（SQL注入、XSS、硬编码密钥等）
2. 边界条件和错误处理
3. 并发安全性
4. 是否有过时或不存在的 API

[粘贴代码]
```

**分角色 Review：**
```
你是一个安全专家，专注于找代码中的安全漏洞，请审查以下代码...

你是一个性能工程师，专注于找代码中的性能问题...

你是一个 Python 资深工程师，检查代码是否符合 PEP8 和最佳实践...
```

## AI 写代码 + AI 写测试

这是目前最高效的工作流之一：

```
Step 1: 让 AI 生成功能代码
"帮我写一个用户注册函数，需要验证邮箱格式和密码强度"

Step 2: 让 AI 生成测试用例（包含边界情况）
"为上面这个函数写 pytest 测试，要覆盖：
- 正常注册
- 邮箱格式错误
- 密码太短
- 密码没有特殊字符
- 重复注册"

Step 3: 跑测试，让测试发现代码问题

Step 4: 把失败的测试发回给 AI 修复
"这个测试失败了：[错误信息]，请修复代码"
```

**为什么有效：**
- AI 知道自己的代码，所以能写出针对性的测试
- 测试用例往往比代码本身更全面
- 形成"代码-测试"闭环，自动发现问题

## 不能无脑信任 AI 的场景

### 🚨 高风险场景，必须人工细审
- **涉及钱的逻辑**：支付、计费、退款
- **权限控制代码**：登录、鉴权、数据隔离
- **数据库 Migration**：字段变更、数据迁移
- **并发 / 分布式**：锁、事务、消息队列
- **密码学相关**：加密、签名、Token 生成

### ✅ 可以放心用 AI 的场景
- 样板代码（CRUD、路由、序列化）
- 正则表达式生成
- 单元测试骨架
- 文档和注释
- 代码格式化和重构

## 团队 AI 代码质量规范建议

```markdown
# 团队 AI 使用规范

## 代码提交要求
- AI 生成的代码必须经过本地测试
- 安全敏感模块（支付、鉴权）禁止直接使用 AI 生成代码
- PR 描述中标注"AI辅助生成"便于 reviewer 重点关注

## Review 重点
- AI 生成代码 review 时间不得少于手写代码
- 安全相关代码需额外 reviewer 确认

## 测试要求
- AI 生成的函数必须有对应测试用例
- 测试覆盖率不低于项目平均水平
```
